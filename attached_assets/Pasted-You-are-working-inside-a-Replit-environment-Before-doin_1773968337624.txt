SECTION 0: REPLIT ENVIRONMENT CAUTIONS — READ BEFORE TOUCHING ANY CODE
═══════════════════════════════════════════════════════════════════════════════

⚠️  CAUTION R-1 — CONTAINER SLEEP & RESTART
  Replit containers sleep after ~5 minutes of inactivity and restart on deploy.
  - NEVER store application state in memory (global variables, in-memory Maps,
    module-level caches). Everything must be persisted in the database.
  - If using JWT / session tokens: store in Supabase auth.sessions or a
    dedicated sessions table — NOT in a module-level Map.
  - After a restart, ALL environment variables reload automatically from
    Replit Secrets. Verify process.env.SUPABASE_URL is read at request-time,
    not cached at startup.
  - ACTION: Before adding any in-memory state, ask: "Will this survive a
    container restart?" If no → use the database.

⚠️  CAUTION R-2 — PORT & PUBLIC URL
  Replit assigns port 3000 (or your configured port) internally. The public
  URL format is: https://<repl-slug>.<replit-username>.repl.co
  - NEVER hardcode this URL anywhere in the codebase.
  - Use process.env.REPLIT_DEV_DOMAIN (Replit injects this automatically) for
    constructing absolute URLs (e.g. email links, OAuth callbacks, CORS origins).
  - CORS must include the Replit domain:
      origin: [`https://${process.env.REPLIT_DEV_DOMAIN}`, 'http://localhost:3000']
  - If the frontend makes fetch() calls, use relative paths ('/api/...') inside
    the same origin, or reference VITE_API_URL set in Replit Secrets.
  - ACTION: Search the codebase for any hardcoded 'repl.co' or 'localhost'
    URLs and replace with environment variable references.

⚠️  CAUTION R-3 — LOCAL FILESYSTEM IS NOT PERSISTENT
  Files written to the local disk (e.g. /tmp/uploads, ./uploads/) will survive
  only within the current container session. On restart or sleep/wake, they
  ARE GONE.
  - ALL file uploads (documents, passports, contracts) MUST go directly to
    Supabase Storage. Do not buffer files to disk then upload — stream to
    Supabase directly using supabase.storage.from('bucket').upload().
  - PDF generation (for reports): generate the Buffer in memory and upload
    directly to Supabase Storage, then return a signed URL. Do NOT write to
    the local filesystem at any point.
  - Exception: the source code directory (project files) IS persistent across
    restarts — only runtime-created files in /tmp and outside the project root
    are at risk.
  - ACTION: Grep for 'fs.writeFile', 'multer({ dest:', 'diskStorage' and
    verify each pipes directly to Supabase or uses memoryStorage.

⚠️  CAUTION R-4 — SECRETS MANAGEMENT
  The .env file is NOT gitignored by default on Replit public repls.
  - ALL secrets (SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY,
    JWT_SECRET, etc.) MUST be stored in Replit's "Secrets" tab (the lock icon),
    NOT in a .env file in the project.
  - The .env file, if it exists, should contain ONLY non-secret placeholders
    or be absent entirely.
  - If the repl is public, add .env to .gitignore immediately.
  - Using SUPABASE_SERVICE_ROLE_KEY on the server is correct for admin ops —
    but NEVER expose it to the browser/client-side code.
  - ACTION: Open the Secrets tab now and verify all keys are there. If .env
    exists and contains real keys, move them to Secrets and clear the file.

⚠️  CAUTION R-5 — CPU / MEMORY THROTTLING
  Free Replit plans have ~0.5 vCPU and 512MB RAM. This affects:
  - PDF generation: use lightweight libraries (pdfkit, pdf-lib). Avoid puppeteer
    (Chromium will exhaust memory). If puppeteer is already in use, switch to
    pdf-lib or pdfkit for this environment.
  - Large file uploads: keep test files under 1MB during testing. Set
    multer or busboy limits to 5MB max.
  - Request timeouts: set server-level timeouts to 120 seconds (Replit's
    reverse proxy cuts at ~120s). Express: server.timeout = 120000.
  - Concurrent API calls: do NOT fire more than 3-4 Supabase calls in
    parallel. Use sequential await chains or Promise.all with max 4 items.
  - ACTION: Add timeout config to the Express server startup if not present.

⚠️  CAUTION R-6 — DATABASE CONNECTION POOLING
  Replit containers restart frequently. Direct PostgreSQL connections (via pg
  or Drizzle with pg) can accumulate stale connections on Supabase's side.
  - Use Supabase's connection pooler (port 6543, Transaction mode) instead of
    direct connection (port 5432) for all server-side DB calls.
  - If using Drizzle ORM or Prisma, ensure DATABASE_URL points to the pooler
    URL (ends in ?pgbouncer=true or uses port 6543).
  - Set pool max to 5 connections (not the default 10+).
  - ACTION: Check DATABASE_URL in Secrets. If port is 5432, switch to 6543
    pooler URL. Add max: 5 to pool config.

⚠️  CAUTION R-7 — ALWAYS SHOW TOOL OUTPUT
  Replit AI sometimes silently skips showing terminal output, server logs, or
  test results. For EVERY test step:
  - Print results with console.log('[TEST RESULT]', ...) so output is
    visible in the Replit console.
  - After each API call in tests, log: status code, response body summary,
    and PASS/FAIL verdict.
  - Do NOT assume a step passed because there was no error. Explicitly assert
    the expected values and log the assertion result.

⚠️  CAUTION R-8 — EMAIL / NOTIFICATION SERVICES
  Replit's outbound SMTP (port 25/465/587) may be blocked on free plans.
  - If the platform uses email notifications (Resend, SendGrid, Nodemailer):
    wrap all email sends in try/catch. A failed email must NOT block the
    main workflow (application approval, contract creation, etc.).
  - For this test run: mock or skip actual email delivery. Log the notification
    payload to console instead: console.log('[NOTIFICATION SKIPPED]', payload).
  - ACTION: Add EMAIL_ENABLED=false to Secrets for the test run, and check
    that the notification service respects this flag.

⚠️  CAUTION R-9 — SEED DATA DEPENDENCY
  Tests reference specific IDs (packageGroupId, userId, etc.) from seed data.
  - Before running any test, verify seed data exists by running the seed script.
  - If the seed script does not exist, create a minimal one covering:

  - ACTION: Run `node seed.js` (or equivalent) and confirm all records exist
    before proceeding to Section 2.
