## New Feature Development — Existing Component Reference Rules

Before creating any new page or component, always follow the steps below.

### 1. Reference Priority

When developing a new feature, always find and read the most closely related
existing component first, then use its structure, patterns, and styling as
the baseline.

| Development Target | Reference File |
|-------------------|---------------|
| Service detail page (tab layout) | The most recently written `*Detail.tsx` in the services folder |
| Service list / cards | `client/src/pages/services/index.tsx` |
| Form component | `*Form.tsx` within the same domain folder |
| Modal | `*Modal.tsx` within the same folder |
| API route | The most structurally similar route file in `server/src/routes/` |
| Drizzle schema | The most similar `_mgt` table in `db/schema.ts` |

### 2. Mandatory Steps Before Writing Code

Before writing any new component or page, you must:
```
① Read the reference file in full
② Identify the import style, hooks, and state management patterns used
③ Identify the tab structure, card layout, and form field patterns
④ Write the new file following exactly the same patterns
```

Never skip these steps. Never introduce a new pattern arbitrarily.

### 3. Consistency Checklist

After completing the file, verify all of the following:

- [ ] Import style matches the reference file (alias, absolute paths, etc.)
- [ ] Tailwind class names follow the same pattern as existing components
- [ ] Tab component uses the same library and approach as existing pages
- [ ] API call method (fetch / axios / react-query) matches existing usage
- [ ] State management approach (useState / useReducer / zustand, etc.) matches existing usage
- [ ] Error handling and loading state patterns match existing usage
- [ ] Design System color values (e.g. `#F5821F`) are used directly —
  never introduce new colors

### 4. Other Service — Tab Structure (Confirmed)

The Others section in the Services page uses exactly the following 4 tabs:

| Order | Tab Name | Primary Content |
|-------|----------|----------------|
| 1 | **Overview** | Service type (Product Group), title, period, assigned staff, partner account, status |
| 2 | **Documents** | File attachments and retrieval (linked to the `files` table) |
| 3 | **Bill** | Billing and cost information, partner cost (AP), Invoice linkage |
| 4 | **Notes** | Memo and comment entry with history log |

> The tab component must follow exactly the same tab implementation used
> in the existing service detail pages — no exceptions.

```
## How This Works in Practice

Once this rule is added, whenever you say **"build the Others tab"**, Claude will automatically follow this sequence:
```
1. Check the full structure of client/src/pages/services/
2. Read one of the existing *Detail.tsx files in full
3. Identify the tab implementation, imports, and styling patterns
4. Write OtherServiceDetail.tsx using the exact same patterns
5. Implement Overview / Documents / Bill / Notes tabs
