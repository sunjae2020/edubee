## 테스트 1 — trust_receipt 타입 Payment 생성

### API 호출

```bash
curl -X POST http://localhost:5000/api/accounting/payments \
  -H "Content-Type: application/json" \
  -d '{
    "payment_type": "trust_receipt",
    "payment_date": "2026-03-25",
    "total_amount": 1000.00,
    "currency": "AUD",
    "payment_method": "bank_transfer",
    "notes": "Bridge E2E Test - trust receipt"
  }'
```

※ 포트 번호가 다르면 실제 포트로 변경
※ 인증 헤더가 필요하면 추가

### 호출 후 즉시 확인

```sql
-- 5분 이내 생성된 레코드만 확인
SELECT 'payment_headers' AS tbl, COUNT(*) AS new_records
FROM payment_headers WHERE created_on > NOW() - INTERVAL '5 minutes'
UNION ALL
SELECT 'transactions', COUNT(*)
FROM transactions WHERE created_on > NOW() - INTERVAL '5 minutes'
UNION ALL
SELECT 'credits', COUNT(*)
FROM credits WHERE created_on > NOW() - INTERVAL '5 minutes'
UNION ALL
SELECT 'journal_entries', COUNT(*)
FROM journal_entries WHERE created_on > NOW() - INTERVAL '5 minutes';
```

### 테스트 1 기대 결과

```
payment_headers  → +1
transactions     → +1   (TRANSACTION-2026-03-25_TRUST_RECEIPT)
credits          → +1   (TRANSACTION-CREDIT-2026-03-25 ...)
debits           → +0   (trust_receipt이므로 debit 없음)
journal_entries  → +1   (기존 로직 유지)
```

## 테스트 2 — trust_transfer 타입 Payment 생성

### API 호출

```bash
curl -X POST http://localhost:5000/api/accounting/payments \
  -H "Content-Type: application/json" \
  -d '{
    "payment_type": "trust_transfer",
    "payment_date": "2026-03-25",
    "total_amount": 500.00,
    "currency": "AUD",
    "payment_method": "bank_transfer",
    "notes": "Bridge E2E Test - trust transfer"
  }'
```

### 호출 후 즉시 확인

```sql
SELECT 'transactions' AS tbl, id, name, payment_description, created_on
FROM transactions WHERE created_on > NOW() - INTERVAL '5 minutes'
UNION ALL
SELECT 'credits', id::text, name, credit_amount::text, created_on
FROM credits WHERE created_on > NOW() - INTERVAL '5 minutes'
UNION ALL
SELECT 'debits', id::text, name, debit_amount::text, created_on
FROM debits WHERE created_on > NOW() - INTERVAL '5 minutes'
ORDER BY created_on DESC;
```

### 테스트 2 기대 결과

```
transactions     → +1   (TRANSACTION-2026-03-25_TRUST_TRANSFER)
credits          → +0   (trust_transfer이므로 credit 없음)
debits           → +1   (TRANSACTION-DEBIT-2026-03-25 ...)
journal_entries  → +1   (기존 로직 유지)
```

## 테스트 3 — 브릿지 실패 내성 확인 (선택)

브릿지 로직 실패 시에도 payment_header 저장이 성공하는지 확인.
(try-catch 처리 확인)

Replit 콘솔에서 서버 로그를 보면서 아래 확인:
- 정상 케이스: `[Bridge]` 관련 에러 로그 없음
- 브릿지 실패 시: `[Bridge] Failed to create transaction record:` 로그 출력
  + payment_header는 정상 저장됨
